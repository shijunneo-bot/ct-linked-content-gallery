About this CleverTap Linked Content field guide

This is a practical field guide with 120 ready-to-run CleverTap Linked Content and Custom KVP campaign examples spanning 15 industries: e-commerce and retail, fintech and banking, food delivery, travel and hospitality, media and streaming, telco, insurance, mobility and ride-hailing, gaming and esports, health and wellness, edtech, real estate and proptech, logistics and supply chain, sports betting and iGaming, and jobs and the gig economy. Each example shows the channel-true message mockup (push, email, SMS, WhatsApp, web push, or webhook), the underlying Liquid personalization, the API and event setup, and a copy-paste campaign recipe you can run on CleverTap. Linked Content pulls live external data into a message at send time; Custom KVP passes key-value pairs the app reads on tap. The guide also includes a developer setup section covering Linked Content configuration, Catalog send-time personalization, Campaign ID personalization, Recommendations, External Trigger campaigns, and Liquid tag hacks such as if/else ladders, default fallbacks, abort, and timezone formatting.

Dev support checklist (hand this to engineering)

  1. Expose or reuse a REST endpoint returning JSON, responding well under 5 seconds at full audience concurrency. CleverTap calls once per recipient and does not retry.
  2. Support GET with auth via header (bearer) or query-param key.
  3. Return UTF-8; add charset=utf-8 for non-Latin scripts.
  4. Keep the response shape stable; CleverTap maps labels to the tested response.
  5. For KVP plays: implement the click handler for your platform and agree a key naming convention with marketing.
  6. Watch your own rate limits: third-party APIs (Radar defaults to 100 req/min) can throttle a large blast before CleverTap does.

What the docs don't say (yet)

Max Linked Content configs per project, response size limit, caching behaviour, and POST support are not published in CleverTap documentation as of July 2026. Treat these as unconfirmed and confirm with your CSM before designing around them. That CSM is us, which is rather the point.

The developer-facing half of this guide: how to stand up Linked Content, Catalog send-time personalization, and Campaign ID tracking, how the pieces stack with External Triggers and Recommendations, and the Liquid patterns that separate a working setup from a resilient one. Written to be handed to an engineering team as-is, or walked through on a TAM call.

⚙️ 1 · Linked Content: full setup Settings → API config → campaign mapping▶

Part A · Create the API configuration

Open the Linked Content setup
Navigate to Settings → Setup → Linked Content and click + Linked Content. To edit an existing one later, use the ellipsis menu.
Name the API
This is a nickname you'll call in Liquid, so shorten http://api.example.com/weather-management/city-weather to just Weather. Short names keep the message templates readable.
Enter the endpoint details
URL, access keys, headers, and query parameters, as documented by the API provider. Auth via header key-value pairs (bearer tokens included) or an access key as a query parameter both work, per the partner recipes.
Personalize the URL itself (optional, powerful)
Both path parameters (mandatory when present in the URL) and query parameters can carry Liquid, and in query params both the keys and the values are personalizable. One endpoint can serve every user, city, and market:
https://api.example.com/weather?location={{ Profile.city }}&unit=m https://movie-api.com/recs?{{ genrename }}={{ genre }}&{{ location }}={{ place }}
Test Linked Content
Click Test Linked Content, supply sample values, and confirm the JSON response comes back in the response box. The response format is always JSON.
Map the objects to labels
Add each object manually or click Auto-fill Objects to pull them from the test response. Rename labels here if useful; these labels are what marketers see at campaign creation.
Test and save changes
Saves the config. If non-English characters garble in responses, append charset=utf-8 to the endpoint, which matters across our APAC scripts.

Part B · Map it into a campaign

Open Personalization Setup
In the campaign's What section, click the personalization icon. The Personalization Setup window opens.
Select your API
Under the Linked Content tab, pick the API(s) this campaign will call.
Map the parameters
Bind each API parameter to a user property, an event property, or a custom value; for example, map the location query param to the profile property Location so each user's call fetches their own city. On External Trigger campaigns, custom field mapping additionally supports Liquid (available when External Trigger is enabled on the account).
Apply, then write
Syntax is Linked.APIName.label, or Linked.["API Name"].label when the name has spaces. Arrays and nested objects walk with dot notation and {% for %} loops.
Send Test Personalization
Always test-send before launch: it exercises the live API call, the mapping, and the Liquid in one pass.
Runtime contract. One real-time call per recipient at send. CleverTap waits 5 seconds and does not retry. On failure or timeout the message still sends, with default: fallbacks (blank where none is set).
Invalid Liquid blocks delivery. Messages with broken syntax are flagged as errors rather than sent with unresolved placeholders, so a typo fails loudly, not embarrassingly.
System labels. raw, json, http_status_code, and headers are always available for health checks and branching.
Scale math. A 500K blast means roughly 500K calls to your endpoint. Size for full audience concurrency, or use Catalog for owned data that tolerates staleness.
TAM talking point: the two questions clients always ask are "what happens if my API is down" (message sends with defaults, never retries, so defaults are mandatory) and "will you hammer my endpoint" (one call per recipient, no caching documented, so yes at audience scale: plan capacity or switch that use case to Catalog).
📦 2 · Catalog Send-Time Personalization event property and user property paths▶

Catalog STP pulls dynamic values (price, category, inventory, restaurant details) from an uploaded catalog at send time. Upload the catalog first (see the Catalog docs), then map it inside a campaign one of two ways.

Path 1 · Personalize by event property

Create a campaign
Any supported channel; the mapping lives per campaign.
Open Personalization → Catalog tab
Click the Personalization link, then the Catalog tab, and select the catalog from the list.
Select the event
The message personalizes off this event's property values. Recent Occurrence rule: the most recent occurrence of the event selects the catalog item; on a triggered campaign, the event that triggered the campaign is the one considered. That rule is what makes abandoned-cart lookups land on the right laptop.
Map event property → catalog column
For example, map Product Name from the Viewed event to the Name column. Repeat for more properties; every additional mapping is an AND condition (name AND color AND brand). Note: catalog mapping is not supported for array-type items in Charged events.
Save, then write the Liquid
{{ Catalog.["Sample Catalog"].Name | default: "Laptop" }} {{ Catalog.["Sample Catalog"].Amount | default: "0" }}

Path 2 · Personalize by user property

Same entry point
Campaign → Personalization → Catalog tab → select the catalog.
Map user property → catalog column
Example: recommend New York's top restaurant, with rich detail from the catalog, to every user whose City property is New York. Multiple mappings AND together here too.
Apply, then write
{{ Catalog.Sample_Catalog.["City Restaurants"] | default:"New York Top Restaurant" }}
Freshness = upload cadence. No endpoint, no timeout risk; the trade is that the data is only as fresh as the last catalog sync. Minute-fresh values belong to Linked Content.
Strings, always. Catalog values read as strings regardless of uploaded type. Do arithmetic server-side before upload.
Power combo. Catalog STP pairs with Constant Event Property for advanced setups where the lookup key needs to be pinned rather than taken from the latest event.
TAM talking point: the decision rule for clients is one sentence: data you own and can upload → Catalog (safe, no endpoint); data that changes minute to minute → Linked Content (live, 5-second window). Most mature accounts run both, split by use case.
🔗 3 · Campaign ID Personalization attribution via deep links and image URLs▶

Campaign ID personalization stamps the campaign's own ID into links and image URLs via Liquid, so downstream systems (attribution partners, web analytics, your backend) can attribute every click and conversion to the exact campaign that caused it.

Deep link / Open URL field
Add the campaign ID as a parameter in the What section's Deep link/Open URL field:
https://demo.app.link/click?ctmessage={{ Campaign.campaignId }} // resolves per send, e.g. ...?ctmessage=123456
Image URL field
Embed the ID in product image URLs so a click on the image itself carries attribution. The docs' example: XYZ Styles tags its Spring Collection with SPR2023 in every featured image URL; each image click links the sale back to that campaign and product.
Close the loop downstream
Whatever reads the URL (attribution partner, backend, GA) now groups traffic and conversions by campaign ID, which is what turns "engagement went up" into "campaign 123456 drove it".
TAM talking point: this is the cheapest attribution win in the platform, and it stacks with Linked Content URL personalization: a single deep link can carry the campaign ID and user context parameters at once (see the hacks below).
✨ 4 · Recommendations & Catalog, the 2.0 pipeline upload → learn → recommend → serve▶
Upload the catalog
CSV with a comma-separated header row; every following row is an item. Two variants: the product catalog, and the product location catalog carrying lat/long per item for distance-filtered plays.
Map columns to events
Multiple catalog columns can map to one event's properties, but all mapped properties must belong to the same event. Values are interpreted as strings.
Let the engine learn
Item relationships form from behaviour on the mapped events; a new recommendation can take up to 24 hours to publish after creation, so build the day before launch.
Create the recommendation
Pick the catalog, the basis event (Viewed Together or Searched Together) with a timeframe (last 10 days, 30 days...), and include/exclude filters on catalog fields (exclude Clearance, include in_stock = true).
Serve in a campaign
Choose the published recommendation, the trigger event (mapped to the catalog), the item sequence (serve items 1 to N), and optionally a location range against the user's last known location. In Past Behavior Segment campaigns, users need the basis event within the last 60 days to qualify.
TAM talking point: the full use-case depth for this pipeline lives in Field Guide 07 (Recommendations & Catalogs), including the Eatigo, Bulbul, and BlueStone public case studies. This section exists so a developer setting up Linked Content sees how the neighbouring machinery bolts on.
🧩 5 · The full stack: External Trigger + Liquid + Catalog + Recommendations how the pieces snap together▶

Each tool answers one question. External Trigger answers when (your backend raises the moment, with a payload). Catalog answers what's true about the item (owned data, keyed off the triggering event via the Recent Occurrence rule). Linked Content answers what's true right now (a live call at send). Recommendations answers what else. Liquid is the glue that reads all four namespaces in one template.

// 1. Your backend raises the moment (External Trigger API), payload included POST /external-trigger { "identity": "user123", "event": "Order Shipped", "fields": { "order_id": "D-2214", "item_id": "SKU-1041", "courier": "NinjaVan" } } // 2. The template reads all namespaces at send: Your {{ Catalog.["retail_products"].name | default:"order" }} is on the move 📦 // Catalog: looked up via the TRIGGERING event's item_id (Recent Occurrence rule) {% if Linked.Courier.eta %}Arriving {{ Linked.Courier.eta }}.{% else %}Tracking inside.{% endif %} // Linked Content: live courier call; ET values can ride the LC URL as params Goes well with it: {% for p in Linked.Recs.items limit:2 %}{{ p.name }} {% endfor %}
ET → Liquid. On External Trigger campaigns, custom field mapping supports Liquid, and trigger values are addressable as {{ ExternalTrigger.somekey }} (also supported inside Linked Content response personalization, currently Private Beta).
ET → Catalog. Because triggered campaigns use the triggering event for the catalog lookup, the payload your backend sends IS the lookup key. Align the ET field names with the mapped event properties and the join is automatic.
ET → Linked Content. Map ET fields into the LC URL parameters and the live call is scoped by the trigger (the corridor, the order, the flight), not just the profile.
The double-check pattern. ET fires on a state change; the LC call re-verifies that state at send. Five cards in the gallery run this (ecom-04, food-05, media-05, edu-01, prop-05); it exists because seconds-old truth can still expire.
TAM talking point: this section is the whiteboard drawing for the technical call. The one-liner: "your backend supplies the moment and the keys; CleverTap supplies the lookup, the live call, the picks, and the send." Clients who get that sentence stop asking which single feature to buy and start designing flows.
🧠 6 · The hacks page Liquid patterns that make setups resilient▶

The if / elsif ladder

Branch entire messages off one live value.
{% if Linked.Weather.temp > 30 %} Sunny in {{ Profile.city | default:"town" }}. BBQ? {% elsif Linked.Weather.temp < 10 %} Cold one. Hot pizza time. {% else %} Order something exotic! {% endif %}
Why: one campaign, three messages, zero extra segments. Also available: unless and case/when for cleaner multi-branch.

default: on every tag

The non-negotiable seatbelt.
{{ Linked.Stock.count | default:"a few" }} left {{ Profile.name | default:"there" }}
Why: LC never retries; on timeout the send proceeds and undefaulted tags render blank. Every snippet in the gallery carries one for exactly this reason.

abort: silence over nonsense

Kill the send when data is missing or wrong.
{% if Linked.Slip.cashout_value %} Cash out now: ${{ Linked.Slip.cashout_value }} {% else %}{% abort %}{% endif %}
Why: a defaulted price is worse than no message. Aborts report as LiquidLogicAborted in campaign stats, so the silence is measurable.

Branch on API health

The system labels are a free circuit breaker.
{% if Linked.API.http_status_code == 200 %} {{ Linked.API.price }} {% else %}{% abort %}{% endif %}
Why: raw, json, http_status_code, headers ship with every config. Status-code gating catches the 500s that a missing-field check misses.

Loops with limit

Top-N lists from array responses.
{% for r in Linked.Rest.restaurants_info limit:5 %} {{ r.name }} · {{ r.user_rating.rating_text }} {% endfor %}
Why: nested dot-walking works inside loops; offset, break, and continue exist for fancier slicing.

date_tz for APAC

One campaign, every timezone honest.
{{ "now" | date_tz: "%Y-%m-%d %H:%M", "Asia/Singapore" }}
Why: plain date returns IST by default; date_tz takes an IANA zone, is DST-aware, and works across channels. Countdown copy across SG, TW, and KR from one template.

assign for readability

Name it once, use it everywhere.
{% assign LANG = Profile.Language %} {% if LANG == "th" %}สวัสดี{% else %}Hello{% endif %}
Why: long templates with repeated deep paths become unreviewable. Assign turns Liquid into something a second engineer can approve.

URL params, keys AND values

One endpoint, every market.
?{{ genrename }}={{ genre }}&country={{ Profile.country }}
Why: both sides of a query param are personalizable, so a single LC config serves every corridor, city, and locale. The fin-03 / telco-03 / travel-03 trick.

Stack Campaign ID into LC links

Attribution and personalization in one URL.
app.link/p/{{ Linked.Match.stream_id }}?ctmessage={{ Campaign.campaignId }}
Why: the deep link routes to the live content and stamps the campaign for attribution. Two doc pages, one line.

Money math stays server-side

Liquid renders; servers calculate.
// API returns the computed figure, print-ready: { "est_monthly": 2380, "rate_pct": 3.10 }
Why: catalog values are strings and template arithmetic is unauditable. Computed finance arrives from the API where compliance can review the formula.

The two-call pattern

Correlate two live sources in one send.
{% if Linked.Weather.raining and Linked.Kitchen.load < 80 %} Rainy night, kitchens ready. 🌧️ {% else %}{% abort %}{% endif %}
Why: multiple LC configs can map into one campaign; each is its own 5-second call, so keep both endpoints hot.

Test like it's production

The pre-launch trio.
1 · Test Linked Content in setup (validates the call). 2 · Send Test Personalization (validates mapping + Liquid on a real device). 3 · Kill your API on purpose and confirm the defaults read acceptably, because one day it will be down for real.
TAM talking point: the hacks worth opening a call with are abort and default:, because they answer the fear question ("what if the data is wrong?") before it's asked. The rest are the difference between a demo and a deployment.
